# VoiceOS Qdrant Configuration Blueprint

This document outlines the proposed Qdrant configuration for the VoiceOS project, specifically for storing and querying vector embeddings related to conversational memory. Qdrant will serve as the vector database, complementing PostgreSQL by enabling semantic search and retrieval-augmented generation (RAG) capabilities.

## Core Principles

*   **Semantic Search**: Enable efficient similarity search for conversational context.
*   **Scalability**: Design collections to handle a growing volume of memory events.
*   **Integration**: Ensure seamless integration with the PostgreSQL `memory_events` table via `embedding_id`.
*   **Performance**: Optimize for low-latency vector search.

## Collection Definitions

### 1. `memory_embeddings` Collection

This collection will store vector embeddings for each conversational memory event, allowing for semantic search and retrieval.

| Parameter | Value | Description |
| :-------- | :---- | :---------- |
| `collection_name` | `memory_embeddings` | Name of the Qdrant collection. |
| `vectors.size` | `[Determined by LLM embedding model]` | Dimensionality of the vector embeddings. This must match the output dimension of the chosen LLM embedding model (e.g., 768 for `text-embedding-ada-002`, 1536 for `text-embedding-3-small`). |
| `vectors.distance` | `Cosine` | Distance metric for vector comparison. Cosine similarity is commonly used for text embeddings. |
| `payload_schema` | `id: Keyword`, `user_id: Integer`, `contact_id: Integer`, `event_type: Keyword`, `timestamp: Datetime` | Schema for the payload associated with each vector. These fields will be used for filtering and linking back to the PostgreSQL `memory_events` table. |
| `on_disk_payload` | `true` | Store payload on disk to reduce memory usage for large collections. |
| `hnsw_config.m` | `16` | Number of connections per node in HNSW graph. |
| `hnsw_config.ef_construct` | `100` | Size of the dynamic list for the HNSW graph. |

## Payload Fields Explained

*   `id`: Corresponds to `memory_events.id` in PostgreSQL. Used to retrieve the full memory event details after a vector search.
*   `user_id`: Corresponds to `memory_events.user_id`. Allows filtering memory events by user.
*   `contact_id`: Corresponds to `memory_events.contact_id`. Allows filtering memory events by contact.
*   `event_type`: Corresponds to `memory_events.event_type`. Allows filtering by the type of conversational event.
*   `timestamp`: Corresponds to `memory_events.timestamp`. Allows time-based filtering of memory events.

## Integration with PostgreSQL

Each entry in the Qdrant `memory_embeddings` collection will have a corresponding record in the PostgreSQL `memory_events` table. The `memory_events.embedding_id` column will store the Qdrant point ID (which can be the same as `memory_events.id` for simplicity, or a UUID generated by Qdrant).

**Workflow for storing a new memory event:**

1.  A new conversational event occurs.
2.  The event content is processed by an LLM embedding model to generate a vector embedding.
3.  The event details (excluding the raw embedding) are stored in the PostgreSQL `memory_events` table, generating a `memory_events.id`.
4.  The vector embedding, along with relevant payload fields (`id`, `user_id`, `contact_id`, `event_type`, `timestamp`), is stored in the Qdrant `memory_embeddings` collection. The Qdrant point ID is set to `memory_events.id`.
5.  The `memory_events.embedding_id` in PostgreSQL is updated with the Qdrant point ID.

**Workflow for retrieving relevant memory events (RAG):**

1.  A query (e.g., user's current message) is processed by the same LLM embedding model to generate a query vector.
2.  A similarity search is performed on the Qdrant `memory_embeddings` collection using the query vector, potentially with payload filters (e.g., `user_id`, `contact_id`).
3.  The top-k nearest neighbor point IDs are retrieved from Qdrant.
4.  These point IDs are used to fetch the full memory event details from the PostgreSQL `memory_events` table.
5.  The retrieved memory events are then used as context for the LLM.

## Future Considerations

*   **Multiple Embedding Models**: If multiple embedding models are used, separate Qdrant collections or multi-vector configurations might be necessary.
*   **Hybrid Search**: Explore combining vector search with keyword search (e.g., using Qdrant's full-text search capabilities on payload fields) for more comprehensive retrieval.
*   **Scalability**: For extremely large datasets, consider Qdrant's distributed deployment options.
*   **Backup and Restore**: Implement regular backup and restore procedures for Qdrant data.
